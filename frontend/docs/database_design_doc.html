Architecture DB - Projet Motion 3D

Notes de travail sur la structure de la base de données.

Les Choix Tech

  • PostgreSQL + JSONB : C'est le plus simple pour gérer les coordonnées 3D (articulations) sans se prendre la tête avec un schéma rigide.
  • Stockage Hybride : Les vidéos restent sur le disque (filesystem), la DB ne garde que le "chemin". On évite de faire exploser la taille de la base.
  • FastAPI & SQLAlchemy : Pour que le code Python parle proprement à la DB.

Le Schéma (Tables)

Utilisateurs (users)

Champ Type Note
user_id Serial (PK) Auto-incrément
pseudo Varchar(50) -
email Varchar(100) Login unique
password Text Haché !
role Enum User / Admin / Coach
profile_json JSONB Poids, taille, objectifs...

Référentiels (movements)

C'est ici qu'on stocke les mouvements "parfaits".

Champ Type Note
name Varchar Nom de l'exo
category Varchar C'est pour filtrer dans l'appli
smpl_ref JSONB Modèle 3D de base
joints_3d JSONB Les 33 points de référence

Performances (performances)

Champ Type Note
user_id Int (FK) Lien vers la table User
video_path Varchar Chemin vers le MP4
score Float % de réussite
results_3d JSONB Les 33 points calculés de l'utilisateur
feedback_txt Text Conseils du coach ou de l'IA

Comment ça circule ?

L'utilisateur upload sa vidéo via Flutter. Le backend FastAPI fait mouliner l'IA (SMPL-X), extrait les 33 joints et calcule le score par rapport au mouvement de référence. Tout est rangé dans le champ JSONB pour pouvoir être ré-affiché plus tard sans refaire les calculs.
